# Plan 17-01: Codegen Service Design + LLM Client Refactoring

## Overview

Design the Codegen service (port 8774) that transforms validated formulas into multi-language code (C99 primary, Rust secondary, Python always) with LLM-generated explanations. Includes refactoring LLM client functions from `analyzer/llm.py` to `shared/llm.py` for reuse.

**Phase type**: Research & Design (deliverable = DESIGN.md)
**Estimated LOC**: 0 production code (design document only)

## Prerequisites

- [x] Phase 16 complete (Validator Testing)
- [x] SymPy codegen API researched (RESEARCH.md)
- [x] LLM explanation prompt design done (RESEARCH.md)
- [x] Service architecture defined (CONTEXT.md)
- [x] `generated_code` table already in schema

## Tasks

### Task 1: LLM Client Refactoring Design

**Goal**: Design the extraction of LLM client functions from `services/analyzer/llm.py` to `shared/llm.py` for reuse by both Analyzer and Codegen.

**Deliverable**: Section in DESIGN.md covering:
- Functions to extract: `call_ollama()`, `call_gemini_sdk()`, `call_gemini_cli()`, `fallback_chain()`, `_get_gemini_api_key()`, `_strip_markdown_fences()`
- New `shared/llm.py` API with configurable fallback order (Gemini-first for Analyzer, Ollama-first for Codegen)
- Backward-compatible import path for Analyzer (update `from services.analyzer.llm import` to `from shared.llm import`)
- Ollama structured output support: `format` parameter for JSON schema enforcement
- `num_ctx` and `keep_alive` configuration parameters

### Task 2: Code Generation Module Design

**Goal**: Design the SymPy-based code generation pipeline (LaTeX → SymPy expr → multi-language code).

**Deliverable**: Section in DESIGN.md covering:
- `services/codegen/generators.py` module structure
- `parse_formula(latex: str) -> sympy.Expr` — LaTeX parsing with ANTLR backend
- `generate_c99(expr, func_name, variables) -> str` — C99 via `codegen("C99")`
- `generate_rust(expr, func_name, variables) -> str` — Rust via `codegen("Rust")`
- `generate_python(expr, func_name, variables) -> str` — Python via `pycode()`
- Variable extraction from SymPy expressions (`expr.free_symbols`)
- Function naming convention (formula_id → function name)
- Error handling for each language generator
- Known limitations and workarounds (Rust integer promotion #26967)

### Task 3: LLM Explanation Module Design

**Goal**: Design the LLM explanation module that generates plain-language descriptions of formulas.

**Deliverable**: Section in DESIGN.md covering:
- `services/codegen/explain.py` module structure
- `explain_formula(latex, context, paper_title) -> FormulaExplanation` function
- System prompt (from RESEARCH.md)
- Ollama `format` parameter with `FormulaExplanation.model_json_schema()`
- Fallback chain: Ollama → Gemini SDK → Gemini CLI (reversed from Analyzer)
- `/no_think` mode for qwen3
- Token budget validation (system ~300 + user ~960 + output ~350 = ~1610, fits 4096 ctx)
- `FormulaExplanation` Pydantic model (in `shared/models.py`)

### Task 4: Service Handler Design

**Goal**: Design the CodegenHandler following the v5.0 Validator pattern.

**Deliverable**: Section in DESIGN.md covering:
- `services/codegen/main.py` — CodegenHandler class
- `POST /process` endpoint: request/response JSON schemas
- Processing pipeline per formula: parse → explain → generate (3 languages) → store
- DB operations: query validated formulas, insert generated_code rows, update formula stage
- Stage transition: `validated` → `codegen` (or `failed`)
- Formula-level error isolation (one failure doesn't block batch)
- Optional `formula_id` filter + `max_formulas` limit + `force` flag
- Configuration: `RP_CODEGEN_PORT`, `RP_CODEGEN_LLM_URL`, `RP_CODEGEN_MAX_FORMULAS`

### Task 5: Schema & Model Updates Design

**Goal**: Document minimal schema changes and new Pydantic models needed.

**Deliverable**: Section in DESIGN.md covering:
- `formulas.description` column usage (store explanation text, already exists)
- `generated_code` table — already in schema, no migration needed
- `FormulaExplanation` Pydantic model addition to `shared/models.py`
- Metadata JSON structure for `generated_code.metadata`: `{function_name, variables, cse_applied, parse_backend}`
- No new tables needed — explanations stored in `formulas.description` field

### Task 6: Write DESIGN.md

**Goal**: Produce the complete design specification for Phase 18 implementation.

**Deliverable**: `DESIGN.md` in phase 17 directory with all sections from Tasks 1-5.

## Risks

| Risk | Mitigation |
|------|------------|
| `parse_latex()` fails on non-standard LaTeX from regex extractor | Log error, skip formula, try simpler re-parsing |
| Rust integer promotion issue (#26967) | Post-process: regex replace bare integers with `f64` casts |
| `antlr4-python3-runtime` version conflict | Pin to 4.11, test in venv before implementation |
| Explanation JSON truncated at 500 tokens | Validate with Pydantic, retry with higher `num_predict` |
