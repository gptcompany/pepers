## 1. Orchestrator Service Architecture
- Port: 8775, follows existing BaseHandler pattern
- Two operational modes: manual (per-paper HTTP) and automatic (cron)
- Talks to services via HTTP localhost:PORT/process

## 2. API Contract
Design these endpoints with exact request/response JSON:

- `POST /run` — Trigger full pipeline or specific stages
  - Request: `{query?: str, stages?: int, paper_id?: int, force?: bool}`
  - If `query` provided: starts from Discovery with these keywords
  - If `paper_id` provided: advances specific paper to next stage
  - `stages`: how many stages to advance (default: all, 1-5)
  - Response: `{run_id: str, status: str, stages_completed: int, results: {...}}`

- `GET /status` — Pipeline overview
  - Response: `{papers_by_stage: {discovered: N, analyzed: N, ...}, last_run: {...}, recent_errors: [...]}`

- `GET /health` — Service health (inherited from BaseHandler)
- `GET /status/services` — Health of all downstream services
  - Calls each service's /health, returns aggregated status

## 3. Orchestration Flow
- Sequential dispatch: calls each service's /process endpoint in order
- Per-paper tracking: records which stage each paper is at
- Batch processing: forwards max_papers/max_formulas to downstream services
- Stage mapping:
  - Discovery: stage=None → discovered (query required)
  - Analyzer: discovered → analyzed/rejected
  - Extractor: analyzed → extracted
  - Validator: extracted → validated
  - Codegen: validated → codegen/complete

## 4. Error Handling & Retry
- Per-paper error isolation: one paper failing doesn't block batch
- Automatic retry: 3 attempts with exponential backoff (1s, 4s, 16s)
- After max retries: mark paper as failed with error message
- Per-stage error tracking: record which stage failed
- Retry only the failed paper, not the entire batch

## 5. Cron Scheduling
- APScheduler BlockingScheduler inside orchestrator process
- Default: daily at 08:00 (configurable via RP_ORCHESTRATOR_CRON)
- Cron behavior: advance all pending papers by configured number of stages
- Config: RP_ORCHESTRATOR_STAGES_PER_RUN (default: 5 = full pipeline)
- Default Discovery query: configurable via RP_ORCHESTRATOR_DEFAULT_QUERY

## 6. Docker Compose Layout
Design docker-compose.yml with:
- 6 services (discovery, analyzer, extractor, validator, codegen, orchestrator)
- Shared named volume `research-data` mounted at /data for SQLite DB + PDFs
- Same user UID across all containers (for WAL SHM file access)
- Base image: python:3.12-slim with multi-stage build
- Health checks using Python urllib against /health endpoints
- depends_on with condition: service_healthy for startup ordering
- network_mode: host for accessing external services (RAGAnything:8767, CAS:8769, Ollama:11434)
  OR extra_hosts for localhost resolution — evaluate and decide
- Environment variables passed via .env file (dotenvx compatible)

## 7. Configuration
List all new RP_ORCHESTRATOR_* environment variables:
- RP_ORCHESTRATOR_PORT=8775
- RP_ORCHESTRATOR_CRON="0 8 * * *"
- RP_ORCHESTRATOR_STAGES_PER_RUN=5
- RP_ORCHESTRATOR_DEFAULT_QUERY="abs:\"Kelly criterion\" AND cat:q-fin.*"
- RP_ORCHESTRATOR_RETRY_MAX=3
- RP_ORCHESTRATOR_RETRY_BACKOFF=4.0

## 8. Directory/File Layout
- services/orchestrator/main.py — Handler + scheduler
- services/orchestrator/pipeline.py — Stage dispatch logic
- services/orchestrator/scheduler.py — APScheduler cron setup
- Dockerfile — Multi-stage build
- docker-compose.yml — All 6 services

Include sequence diagrams (ASCII) for:
- Manual trigger flow (POST /run with paper_id)
- Cron batch flow (advance all pending)
- Error/retry flow</action>
