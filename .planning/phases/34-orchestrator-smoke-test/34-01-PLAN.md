3. **New function `run_smoke_test_via_orchestrator()`**:
```python

    arxiv_id: str = DEFAULT_ARXIV_ID,
    db_path: Path = DEFAULT_DB_PATH,
    max_formulas: int = 50,
) -> SmokeReport:
```

Flow:
- Health check: GET orchestrator:8775/health
- Service health: GET orchestrator:8775/status/services → verify all_healthy
- POST /run with `{"query": "id:{arxiv_id}", "stages": 5, "max_papers": 1, "max_formulas": max_formulas}`
- Map response to SmokeReport:
  - `passed = response["status"] == "completed" and response["stages_completed"] == 5`
  - Create StepResult entries from `response["results"]` dict
  - Verify DB stage = "codegen" for the paper
  - Collect formula stats with existing `_formula_counts()`

4. **Update `main()`** to dispatch:
```python
if args.via_orchestrator:
    report = run_smoke_test_via_orchestrator(args.arxiv_id, db_path, args.max_formulas)
else:
    report = run_smoke_test(args.arxiv_id, db_path, args.max_formulas)
```

```markdown
# Research Pipeline — Operational Runbook

## 1. Service Overview
| Service | Port | Description | External Dependencies |
| discovery | 8770 | arXiv/S2/CrossRef paper search | arXiv API, Semantic Scholar, CrossRef |
| analyzer | 8771 | LLM relevance scoring | Ollama/OpenRouter/Gemini |
| extractor | 8772 | PDF → formula extraction | RAGAnything (:8767) |
| validator | 8773 | CAS mathematical validation | CAS engine (:8769), SymPy |
| codegen | 8774 | Formula → code generation | Ollama/OpenRouter |
| orchestrator | 8775 | Pipeline coordination | All above services |

## 2. Prerequisites
- Python 3.11+, SQLite 3.35+
- RAGAnything server on port 8767
- CAS validation server on port 8769
- At least one LLM provider (Ollama recommended for local)
- dotenvx for secret management

## 3. Startup Order
1. External deps: RAGAnything, CAS, Ollama
2. Pipeline services (order matters for health checks):
   discovery → analyzer → extractor → validator → codegen → orchestrator

### systemd
systemctl start research-pipeline.target  # starts all in dependency order

### Docker Compose
docker compose up -d

### Manual
python -m services.discovery.main &
# ... etc

## 4. Health Check URLs
GET http://localhost:{port}/health for each service
Expected response: {"status": "ok", "service": "...", "uptime_seconds": N, ...}
Orchestrator also: GET /status/services → {"all_healthy": bool, "services": {...}}

## 5. Common Failure Modes

### 5.1 Service Crash During Batch Processing
- Symptom: Service PID gone, formulas stuck at intermediate stage
- Cause: OOM, uncaught exception, kill -9
- Impact: Paper stage may be inconsistent
- Recovery: systemd auto-restarts (Restart=always), consistency check runs at startup

### 5.2 SQLite WAL Contention
- Symptom: "database is locked" errors in logs
- Cause: Multiple services writing simultaneously
- Recovery: Usually resolves automatically; if persistent, stop all services, delete WAL/SHM files

### 5.3 External API Rate Limits
- arXiv: 3 req/sec → automatic 1s delay in discovery
- Semantic Scholar: 100 req/5min → retry with backoff
- CrossRef: generous limits, rarely hit
- Recovery: Wait and retry, or reduce max_papers parameter

### 5.4 CAS Engine Timeout
- Symptom: Validator takes >17s/formula average
- Cause: Complex LaTeX, SymPy/Maxima hanging on edge cases
- Recovery: formula marked "failed", pipeline continues with next batch

### 5.5 LLM Provider Unreachable
- Symptom: Analyzer/codegen return connection errors
- Cause: Ollama not running, OpenRouter key expired, Gemini quota
- Recovery: fallback_chain tries next provider automatically

### 5.6 Extractor PDF Processing Failure
- Symptom: MinerU/RAGAnything timeout or crash
- Cause: Corrupted PDF, huge file, GPU OOM
- Recovery: Paper marked "failed", skip to next

## 6. Recovery Procedures

### 6.1 Paper Stuck in Intermediate Stage
SELECT id, arxiv_id, stage, error FROM papers WHERE stage NOT IN ('codegen', 'rejected', 'failed');
-- For stuck paper: reset and reprocess
UPDATE papers SET stage = 'discovered', error = NULL WHERE id = ?;
-- Then trigger via orchestrator:
curl -X POST localhost:8775/run -d '{"paper_id": ID, "stages": 5}'

### 6.2 Duplicate Formula Detection
-- Schema v3 UNIQUE(paper_id, latex_hash) prevents duplicates
-- If duplicates exist from pre-v3:
SELECT paper_id, latex_hash, COUNT(*) FROM formulas GROUP BY paper_id, latex_hash HAVING COUNT(*) > 1;

### 6.3 Batch Iteration Safety Cap
-- Orchestrator caps at 100 iterations, smoke_test at 100
-- If cap hit, check for formulas cycling between stages:
SELECT stage, COUNT(*) FROM formulas WHERE paper_id = ? GROUP BY stage;

### 6.4 Full Pipeline Re-run
curl -X POST localhost:8775/run -d '{"query": "id:ARXIV_ID", "stages": 5, "force": true}'

| Variable | Default | Description |
| RP_DB_PATH | data/research.db | SQLite database path |
| RP_HOST | 0.0.0.0 | Bind address |
| RP_DISCOVERY_PORT | 8770 | Discovery service port |
| ... (all 6 service ports) |

| RP_ORCHESTRATOR_RETRY_BACKOFF | 4.0 | Exponential backoff base |
| RP_LLM_TEMPERATURE | 0 | LLM temperature (0=deterministic) |
| RP_EXTRACTOR_TIMEOUT | 3600 | Extractor timeout for CPU |
| RP_MAX_FORMULAS | 50 | Batch size for validator/codegen |

## 8. Extractor Performance Notes
- CPU (Intel i7): ~10 min/page for RAGAnything PDF processing
- GPU (NVIDIA): ~1 min/page
- First run downloads models (~2GB) — subsequent runs use cache
- Recommended RP_EXTRACTOR_TIMEOUT: CPU=3600, GPU=600
- Memory: 4GB+ RAM for PDF processing

## 9. Monitoring
- Logs: journalctl -u research-{service}.service -f
- Pipeline status: GET localhost:8775/status
- Service health: GET localhost:8775/status/services
- DB stats: sqlite3 data/research.db "SELECT stage, COUNT(*) FROM papers GROUP BY stage"
```
