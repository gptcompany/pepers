# Phase 46, Plan 01: process-exporter + Prometheus scrape + alert rules

## Goal

Configure the monitoring stack to scrape PePeRS metrics, name PePeRS processes distinctly in process-exporter, and alert on service failures.

## Prerequisites

- Phase 44 complete (all services expose /metrics)
- Monitoring stack running: VictoriaMetrics (:8428), process-exporter (:9256), Alertmanager (:9093)

## Approach

Modify monitoring-stack config files directly (same machine, same operator). This is the standard ops pattern — monitoring-stack owns its configs, we add PePeRS entries. No deploy.sh needed.

Copy to /etc/prometheus/ and reload services requires sudo (one-time admin action after each config change, same pattern as all other monitoring changes).

## Tasks

### Task 1: process-exporter config

**File:** `/media/sam/1TB/monitoring-stack/process-exporter/config.yml`

Add PePeRS entry **BEFORE** the generic Python catchall. Verified cmdline from host `ps aux`: `python -m services.discovery.main`

```yaml
# PePeRS microservices (Docker, host network, init: true)
- name: "pepers-{{.Matches.svc}}"
  cmdline:
  - 'python\s+-m\s+services\.(?P<svc>discovery|analyzer|extractor|validator|codegen|orchestrator)\.main'
```

This matches exactly `python -m services.{name}.main` and produces named groups: `pepers-discovery`, `pepers-analyzer`, etc.

**Deploy:** `sudo systemctl restart process-exporter`

### Task 2: Prometheus scrape config

**File:** `/media/sam/1TB/monitoring-stack/prometheus/prometheus.yml`

Append PePeRS scrape job:
```yaml
# PePeRS Pipeline Services (Phase 44 /metrics endpoints)
- job_name: 'pepers'
  honor_labels: true
  static_configs:
    - targets: ['localhost:8770', 'localhost:8771', 'localhost:8772', 'localhost:8773', 'localhost:8774', 'localhost:8775']
      labels:
        machine: 'workstation'
        env: 'local'
```

`honor_labels: true` preserves PePeRS' own `service` label (discovery, analyzer, etc.).

**Deploy:** `sudo cp /media/sam/1TB/monitoring-stack/prometheus/prometheus.yml /etc/prometheus/prometheus.yml && sudo systemctl restart victoria-metrics`

### Task 3: Alert rules

**File:** `/etc/prometheus/alert.rules.yml`

Add `pepers_alerts` group:
```yaml
- name: pepers_alerts
  rules:
    - alert: PepersServiceDown
      expr: up{job="pepers"} == 0
      for: 3m
      labels:
        severity: critical
      annotations:
        summary: "PePeRS service down ({{ $labels.instance }})"
        description: "PePeRS service at {{ $labels.instance }} is not responding for 3+ minutes"

    - alert: PepersNoPapersProcessed
      expr: increase(pepers_papers_processed_total[24h]) == 0
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "No papers processed in 24 hours"
        description: "PePeRS pipeline has not processed any papers in the last 24 hours"
```

Note: `for: 5m` is short — the `increase[24h]` already covers the window. The `for` just debounces to avoid firing during initial startup.

**Deploy:** VictoriaMetrics auto-reloads rules files on change (inotify), or `kill -HUP` to force.

### Task 4: Verification

After deploying all configs and starting PePeRS containers:
1. `curl -s localhost:9256/metrics | grep 'pepers-'` — process-exporter named groups
2. `curl -s localhost:8428/api/v1/targets | grep pepers` — scrape targets UP
3. `curl -s localhost:8428/api/v1/rules | grep -c Pepers` — alert rules loaded (should be 2)
