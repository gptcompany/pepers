## Step-by-Step Execution

### Step 0: Verifica Infrastruttura
```bash
for port in 8767 8769 8770 8771 8772 8773 8774 8775; do
  echo -n "Port $port: "
  curl -s --max-time 5 http://localhost:$port/health | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('service','?'), d.get('status','?'))" 2>/dev/null || echo "DOWN"
done
curl -s http://localhost:11434/api/tags | python3 -c "import sys,json; models=json.load(sys.stdin).get('models',[]); print(f'{len(models)} models:', [m['name'] for m in models])"
```
**Gate**: Tutti i servizi devono rispondere "ok". Se qualcuno è DOWN, risolvere prima di procedere.

### Step 0b: Confidence Gate — Infrastruttura
Passare i risultati del health check nel confidence gate per verificare completezza.
Analizzare: tutti i servizi rispondono? Volume permissions OK? Ollama ha i modelli giusti?

### Step 1: Discovery (:8770)
```bash
curl -s --max-time 120 -X POST http://localhost:8770/process \
  -H "Content-Type: application/json" \
  -d '{"query": "abs:\"Kelly criterion\" AND cat:q-fin.*", "max_results": 3}'
```

**Verifiche**:
- [ ] Response status 200 con `papers_found > 0`
- [ ] Papers inseriti nel DB (`SELECT * FROM papers WHERE stage='discovered'`)
- [ ] Campi popolati: title, abstract, arxiv_id, authors
- [ ] Enrichment S2/CrossRef: citation_count, doi presenti (o errori documentati)
- [ ] `time_ms` ragionevole (< 30s per 3 papers)

**Metriche**: papers_found, papers_new, enrichment_success_rate, time_ms

### Step 1b: Confidence Gate — Discovery
Analizzare gaps e criticità:
- Qualità dati: titoli sensati? Abstract presente?
- Enrichment: S2 e CrossRef funzionano? Rate limiting?
- Coverage: i papers trovati sono rilevanti al Kelly criterion?
- Performance: tempo accettabile per batch giornaliero?

---

### Step 2: Analyzer (:8771)
```bash
curl -s --max-time 600 -X POST http://localhost:8771/process \
  -H "Content-Type: application/json" \
  -d '{"max_papers": 3}'
```

**Verifiche**:
- [ ] Response 200 con papers analizzati
- [ ] LLM scoring effettivo (non fallback vuoto)
- [ ] Quale LLM usato? (Ollama qwen3:8b? Gemini? fallback chain?)
- [ ] Score presenti per ogni paper
- [ ] Threshold 0.7: papers ABOVE → stage='analyzed', papers BELOW → stage='rejected'
- [ ] prompt_version registrato
- [ ] `time_ms` per paper (aspettativa: 5-30s per Ollama, 2-10s per Gemini)

**Metriche**: papers_analyzed, papers_rejected, avg_score, llm_used, time_ms_per_paper

### Step 2b: Confidence Gate — Analyzer
Analizzare gaps e criticità:
- Score ragionevoli? Papers filosofici filtrati vs papers finanziari?
- LLM usato: Ollama è lento? Gemini disponibile?
- Threshold 0.7 discrimina bene?
- Falsi positivi/negativi: papers non-Kelly sopra threshold?

### Step 3: Extractor (:8772)
```bash
curl -s --max-time 1800 -X POST http://localhost:8772/process \
  -H "Content-Type: application/json" \
  -d '{"max_papers": 3}'
```

**Verifiche**:
- [ ] PDF scaricato da arXiv (check file in /data/pdfs/)
- [ ] RAGAnything chiamato e risposta ricevuta (o timeout documentato)
- [ ] Path mapping container→host funziona (RP_EXTRACTOR_PDF_HOST_DIR)
- [ ] Formule LaTeX estratte (`SELECT * FROM formulas WHERE paper_id IN (...)`)
- [ ] Formula di Kelly presente? (`f^* = (bp - q) / b` o variante)
- [ ] Context estratto (200 char intorno alla formula)
- [ ] Dedup hash funziona (no duplicati)
- [ ] `time_ms` (aspettativa: 15-60 min per paper con MinerU su CPU)

**Metriche**: pdfs_downloaded, formulas_extracted, unique_formulas, rag_time_ms, total_time_ms

### Step 3b: Confidence Gate — Extractor
Analizzare gaps e criticità:
- RAGAnything performance: MinerU tempo per paper? Bottleneck?
- Qualità estrazione: formule LaTeX corrette? Rumore?
- Path mapping: fix Docker→host funziona?
- Edge cases: PDF grandi, PDF con molte pagine, PDF senza formule

### Step 4: Validator (:8773)
```bash
curl -s --max-time 300 -X POST http://localhost:8773/process \
  -H "Content-Type: application/json" \
  -d '{"max_formulas": 10}'
```

**Verifiche**:
- [ ] SymPy engine chiamato
- [ ] Maxima engine chiamato
- [ ] MATLAB engine chiamato (se disponibile)
- [ ] Consensus: quante VALID, INVALID, PARTIAL, UNPARSEABLE?
- [ ] Fallback funziona? (se un engine errora, gli altri danno consensus)
- [ ] `time_ms` per engine
- [ ] Validations table popolata (`SELECT * FROM validations`)
- [ ] Formula stage aggiornato a 'validated' dove consensus != UNPARSEABLE

**Metriche**: formulas_valid, formulas_invalid, formulas_partial, formulas_unparseable, engine_times, consensus_rate

### Step 4b: Confidence Gate — Validator
Analizzare gaps e criticità:
- MATLAB è bottleneck? Timeout frequenti?
- Consensus ragionevole? Engines d'accordo?
- Formule UNPARSEABLE: sono davvero non parsabili o bug nel parser?
- Performance: tempo per formula

### Step 5: Codegen (:8774)

### Step 5b: Confidence Gate — Codegen

- Error isolation: un linguaggio fallisce, gli altri OK?
- Performance: tempo per formula
