# Plan 23-01: Full Pipeline E2E Smoke Test with Real Papers

**Phase 23 — Post-v7.0 Validation**
**Goal**: Verify every service works end-to-end with real data, honest assessment of bugs/performance/gaps

## Prerequisiti

Servizi che DEVONO essere attivi prima di iniziare:
- CAS microservice (:8769) — SymPy + Maxima + MATLAB
- RAGAnything (:8767) — text extraction
- Ollama (:11434) — LLM (qwen3:8b)

Servizi della pipeline (avviare manualmente o via Docker):
- Discovery (:8770)
- Analyzer (:8771)
- Extractor (:8772)
- Validator (:8773)
- Codegen (:8774)
- Orchestrator (:8775)

## Test Papers

Selezionare 2 papers arXiv corti (2-3 pagine, formule matematiche chiare):

| Paper | Criterio | Perché |
|-------|----------|--------|
| Paper A | Kelly criterion / formula di Kelly | Core domain del progetto, formule note e verificabili |
| Paper B | Formula matematica semplice (Black-Scholes, Sharpe ratio, o simile) | Benchmark alternativo, formule diverse |

**Ricerca papers**: `curl "http://export.arxiv.org/api/query?search_query=all:kelly+criterion&max_results=5"` — scegliere il più corto con formule esplicite.

## Step-by-Step Execution

### Step 0: Verifica Infrastruttura
```bash
# Health check di TUTTI i servizi
for port in 8767 8769 8770 8771 8772 8773 8774 8775; do
  echo -n "Port $port: "
  curl -s http://localhost:$port/health | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('service','?'), d.get('status','?'))" 2>/dev/null || echo "DOWN"
done

# Verifica Ollama
curl -s http://localhost:11434/api/tags | python3 -c "import sys,json; models=json.load(sys.stdin).get('models',[]); print(f'{len(models)} models:', [m['name'] for m in models])"
```
**Gate**: Tutti i servizi devono rispondere "ok". Se qualcuno è DOWN, risolvere prima di procedere.
**Confidence gate**: Verificare con PAL che l'infrastruttura sia completa.

### Step 1: Discovery (:8770)
```bash
# Inserire papers nel DB via discovery service
curl -s -X POST http://localhost:8770/process \
  -H "Content-Type: application/json" \
  -d '{"keywords": "kelly criterion", "max_results": 2}'
```

**Verifiche**:
- [ ] Response status 200 con `papers_found > 0`
- [ ] Papers inseriti nel DB (`SELECT * FROM papers WHERE stage='discovered'`)
- [ ] Campi popolati: title, abstract, arxiv_id, authors
- [ ] Enrichment S2/CrossRef: citation_count, doi presenti (o errori documentati)
- [ ] `time_ms` ragionevole (< 30s per 2 papers)

**Metriche da registrare**: papers_found, papers_new, enrichment_success_rate, time_ms

**Confidence gate**: Discutere qualità dei dati inseriti, coverage enrichment.

### Step 2: Analyzer (:8771)
```bash
# Analizzare i papers discovered
curl -s -X POST http://localhost:8771/process \
  -H "Content-Type: application/json" \
  -d '{"max_papers": 2}'
```

**Verifiche**:
- [ ] Response 200 con papers analizzati
- [ ] LLM scoring effettivo (non fallback vuoto)
- [ ] Quale LLM usato? (Ollama qwen3:8b? Gemini? fallback chain?)
- [ ] Score 5 criteri presenti: kelly_relevance, mathematical_rigor, novelty, practical_applicability, data_quality
- [ ] Threshold 0.7: papers ABOVE → stage='analyzed', papers BELOW → stage='rejected'
- [ ] prompt_version registrato
- [ ] `time_ms` per paper (aspettativa: 5-30s per Ollama, 2-10s per Gemini)

**Metriche**: papers_analyzed, papers_rejected, avg_score, llm_used, time_ms_per_paper

**Confidence gate**: Valutare se gli score sono sensati per papers Kelly criterion.

### Step 3: Extractor (:8772)
```bash
# Estrarre formule dai papers analizzati
curl -s -X POST http://localhost:8772/process \
  -H "Content-Type: application/json" \
  -d '{"max_papers": 2}'
```

**Verifiche**:
- [ ] PDF scaricato da arXiv (check file in /data/pdfs/)
- [ ] RAGAnything chiamato e risposta ricevuta (o timeout documentato)
- [ ] Formule LaTeX estratte (`SELECT * FROM formulas WHERE paper_id IN (...)`)
- [ ] Formula di Kelly presente? (`f^* = (bp - q) / b` o variante)
- [ ] Context estratto (200 char intorno alla formula)
- [ ] Dedup hash funziona (no duplicati)
- [ ] `time_ms` (aspettativa: 30-120s per PDF download + RAG processing)

**Metriche**: pdfs_downloaded, formulas_extracted, unique_formulas, rag_time_ms, total_time_ms

**Confidence gate**: Valutare qualità estrazione LaTeX — le formule sono corrette? Mancano formule importanti?

### Step 4: Validator (:8773)
```bash
# Validare le formule estratte con 3 engine CAS
curl -s -X POST http://localhost:8773/process \
  -H "Content-Type: application/json" \
  -d '{"max_formulas": 10}'
```

**Verifiche**:
- [ ] MATLAB engine chiamato (primo nella lista)
- [ ] SymPy engine chiamato
- [ ] Maxima engine chiamato
- [ ] Consensus: quante VALID, INVALID, PARTIAL, UNPARSEABLE?
- [ ] Fallback funziona? (se MATLAB errora, SymPy+Maxima danno consensus)
- [ ] `time_ms` per engine (aspettativa: MATLAB ~20s, SymPy <1s, Maxima <1s)
- [ ] Validations table popolata (`SELECT * FROM validations`)
- [ ] Formula stage aggiornato a 'validated' dove consensus != UNPARSEABLE

**Metriche**: formulas_valid, formulas_invalid, formulas_partial, formulas_unparseable, engine_times, consensus_rate

**Confidence gate**: MATLAB è il bottleneck? Il consensus è ragionevole? Formule UNPARSEABLE sono davvero non parsabili?

### Step 5: Codegen (:8774)
```bash
# Generare codice dalle formule validate
curl -s -X POST http://localhost:8774/process \
  -H "Content-Type: application/json" \
  -d '{"max_formulas": 10}'
```

**Verifiche**:
- [ ] LLM explanation generata (plain-language)
- [ ] Codice C99 generato (via SymPy codegen())
- [ ] Codice Rust generato (via SymPy codegen())
- [ ] Codice Python generato (via SymPy pycode())
- [ ] Per-language error isolation funziona (un linguaggio fallisce, gli altri continuano)
- [ ] `time_ms` (aspettativa: 5-30s per formula, dipende da LLM)
- [ ] Il codice generato è sintatticamente valido?
- [ ] Formula stage aggiornato a 'codegen'

**Metriche**: formulas_processed, languages_generated, explanation_quality, time_ms_per_formula

**Confidence gate**: Il codice generato è utilizzabile? L'explanation ha senso? Parse errors?

### Step 6: Orchestrator Full Run (:8775)
```bash
# Test orchestrator end-to-end (dovrebbe chiamare tutti i servizi in sequenza)
curl -s -X POST http://localhost:8775/run \
  -H "Content-Type: application/json" \
  -d '{"keywords": "sharpe ratio formula", "max_results": 1}'
```

**Verifiche**:
- [ ] Orchestrator chiama tutti i 5 servizi in sequenza
- [ ] Retry logic funziona (se un servizio è lento)
- [ ] run_id generato e tracciabile
- [ ] Pipeline status (`GET /status`) mostra risultati
- [ ] Services health (`GET /status/services`) mostra tutti healthy
- [ ] `time_ms` totale pipeline (aspettativa: 2-5 minuti per 1 paper)
- [ ] Errori propagati correttamente (non swallowed silently)

**Metriche**: total_time_ms, stages_completed, papers_through_pipeline, errors

**Confidence gate**: Il flusso end-to-end funziona davvero? Performance accettabili?
