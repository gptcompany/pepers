# Plan 28-01: Fix Stage Transitions + Batch Overflow

## Phase

28 — Fix Stage Transitions + Batch Overflow (v9.0 Pipeline Hardening)

## Goal

Fix 4 bugs that prevent the pipeline from advancing papers through stages. After this plan, a paper should flow: extracted → validated → codegen → complete, and all formulas should be processed regardless of batch size.

## Bugs

| # | Severity | File | Bug |
|---|----------|------|-----|
| 1 | CRITICAL | `services/validator/main.py` | No `papers.stage` update after validation — paper stays "extracted" |
| 2 | CRITICAL | `services/codegen/main.py` | No `papers.stage` update after codegen — paper stays "validated" |
| 3 | HIGH | `services/orchestrator/pipeline.py` | Single call per stage — formulas beyond `batch_size=50` never processed |
| 4 | MEDIUM | `shared/llm.py` | OpenRouter `max_tokens=500` truncates responses (should be 4096) |

## Changes

### Bug 1: Validator — Add `_update_paper_stage()` (validator/main.py)

**What**: After processing all formulas for a paper, update `papers.stage` to `'validated'`.

**How**:
1. Add function `_update_paper_stage(db_path, paper_id, stage)` that runs `UPDATE papers SET stage = ?, updated_at = datetime('now') WHERE id = ?`
2. After the formula processing loop (line ~154), collect unique `paper_id` values from processed formulas
3. For each paper_id, update `papers.stage = 'validated'` only if at least one formula succeeded (not all failed)

**Edge cases**:
- If `paper_id` was passed in request: update just that paper
- If batch mode (no paper_id): collect paper_ids from processed formulas, update each
- If ALL formulas for a paper failed: don't advance paper stage (leave as-is)

### Bug 2: Codegen — Add `_update_paper_stage()` (codegen/main.py)

**What**: Same pattern as validator — after processing formulas, update `papers.stage` to `'codegen'`.

**How**:
1. Add function `_update_paper_stage(db_path, paper_id, stage)` (same as validator)
2. After the formula processing loop (line ~153), collect unique paper_ids
3. For each paper_id, update `papers.stage = 'codegen'` if at least one formula had successful codegen

**Edge cases**: Same as Bug 1.

### Bug 3: Orchestrator — Add batch iteration loop (orchestrator/pipeline.py)

**What**: For validator and codegen stages, loop until service reports `formulas_processed == 0` (no more formulas to process).

**How**:
1. In `PipelineRunner.run()`, after calling `_call_service_with_retry()` for validator/codegen stages, check if `result["formulas_processed"] > 0`
2. If yes, re-call the same stage (same params) in a while loop
3. Break when `formulas_processed == 0` (all formulas in this stage have been processed)
4. Add safety cap: max 100 iterations to prevent infinite loop

**Why validator/codegen only**: Discovery, Analyzer, Extractor operate on papers (not formulas), and their batch handling is already correct.

### Bug 4: OpenRouter `max_tokens` (shared/llm.py)

**What**: Increase `max_tokens` from 500 to 4096 to match Gemini SDK.

**How**:
1. Line 171: Change `"max_tokens": 500` → `"max_tokens": 4096`
2. Line 220: Change Ollama `"num_predict": 500` → `"num_predict": 4096` (same issue)

## Risks

- **Stage update timing**: Papers should only advance when formulas are actually processed, not when batch is empty (0 formulas). The current response already returns `formulas_processed` count — we use this as guard.
- **Infinite loop**: Batch iteration needs safety cap. 100 iterations × 50 formulas = 5000 formulas per paper, more than sufficient.
- **Backward compat**: Adding paper stage update is additive — existing formula-level logic unchanged.

## Success Criteria

- [ ] `papers.stage` advances from `extracted` → `validated` after validator processes formulas
- [ ] `papers.stage` advances from `validated` → `codegen` after codegen processes formulas
- [ ] All formulas processed when count > batch_size (verified by loop count > 1)
- [ ] OpenRouter/Ollama responses not truncated (max_tokens=4096)
- [ ] Existing tests pass (no regressions)
