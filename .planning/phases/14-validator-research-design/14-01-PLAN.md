# Plan 14-01: Validator Service Design + CAS Microservice Architecture

## Overview

Design the Validator service (port 8773) and a standalone CAS microservice (separate repo, port 8769). The CAS service handles SymPy + Maxima engine execution. The Validator orchestrates formula validation with consensus logic.

**Phase type**: Research & Design (deliverable = DESIGN.md)
**Estimated LOC**: 0 production code (design document only)

## Prerequisites

- [x] Phase 13 complete (Extractor Testing)
- [x] CAS microservice analysis done (RESEARCH.md)
- [x] SymPy/LaTeX parsing research done
- [x] User decision: separate CAS repo

## Tasks

### Task 1: CAS Microservice Design

**Goal**: Design the standalone CAS service API, engine architecture, and repo structure.

**Deliverable**: Section in DESIGN.md covering:
- Repo structure (`cas-service/`)
- API contract: `POST /validate`, `GET /health`, `GET /status`, `GET /engines`
- Request/response schemas
- Engine abstraction: base class + SymPy impl + Maxima impl
- LaTeX preprocessing pipeline (strip envs, normalize commands)
- Error handling (parse failures, engine timeouts, subprocess crashes)
- Configuration via env vars (port, maxima path, timeout)
- systemd unit file design

### Task 2: Validator Service Design

**Goal**: Design the Validator microservice that calls CAS service and applies consensus.

**Deliverable**: Section in DESIGN.md covering:
- File structure (`services/validator/`)
- ValidatorHandler with `/process` endpoint
- CAS client module (HTTP calls to CAS service)
- Consensus logic module (2-engine agreement)
- DB operations (read formulas, write validations, update stages)
- Validation report format
- Configuration via RP_VALIDATOR_* env vars
- Error handling matrix

### Task 3: Data Flow & Schema Validation

**Goal**: Verify existing DB schema supports the Validator service.

**Deliverable**: Section in DESIGN.md covering:
- Input: `formulas` table where `stage='extracted'`
- Output: `validations` table (one row per formula per engine)
- Stage transitions: `extracted` → `validated` | `failed`
- No schema migration needed (tables already exist)

### Task 4: Write DESIGN.md

**Goal**: Produce the complete design specification.

**Deliverable**: `DESIGN.md` in phase 14 directory, ready for Phase 15 implementation.

## Acceptance Criteria

- [ ] DESIGN.md covers both CAS microservice and Validator service
- [ ] API contracts are fully specified (request/response JSON)
- [ ] File structures defined for both repos
- [ ] Consensus logic algorithm documented
- [ ] LaTeX preprocessing steps documented
- [ ] Error handling matrix complete
- [ ] No ambiguity for Phase 15 implementation

## Risks

| Risk | Mitigation |
|------|------------|
| SymPy parse_latex has silent failures | Design includes parse error detection + logging |
| Maxima subprocess may hang | Timeout + SIGKILL fallback in CAS service |
| Formula complexity varies widely | Design includes "unparseable" category (not failed) |
